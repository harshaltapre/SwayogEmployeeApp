# SWAYOG Partner Mobile Application
## Technical Specification & UI/UX Design Blueprint

This document defines the architecture, user experience (UI/UX) design system, screen flows, and database API synchronization mechanism for the **SWAYOG Partner Mobile Application**. The application is designed to help local business partners, solar distributors, and sales agents register new solar projects, track referral stages, audit commission structures, withdraw earnings, and communicate directly with the SWAYOG admin staff.

---

## 1. Executive Summary & App Concept

*   **Goal**: Create a distinct, business-focused Android application that connects partners directly to the central dashboard databases.
*   **Target Platform**: Android (Kotlin & Jetpack Compose or Flutter).
*   **Core Objectives**:
    1.  **Lead Generation & Referral Intake**: Mobile forms to submit client leads directly from the field.
    2.  **Project Lifecycle Tracker**: Real-time progress monitoring (Lead -> Site Survey -> Installation -> Complete).
    3.  **Financial Transparency**: Clear commission ledger showing paid vs. pending referral payments.
    4.  **Earnings Withdrawal**: Trigger payout requests directly into bank accounts.
    5.  **Live Support Chat**: WebSocket/SSE chat functionality linking partners to SWAYOG support admins.
    6.  **Offline Lead Capture**: Enable partners to capture customer leads in remote areas without internet, queuing them for automatic upload.

## 3. Authentication & Login Flow

```
[Partner Opens App] -> [Token Validation]
                            |-- Valid -> [Enter Partner Dashboard]
                            |-- Invalid -> [Partner Login Screen]
```

### A. Login & Onboarding
*   **Sign-In Input**:
    *   `Partner Login ID` or `Email`.
    *   `Password` (secured input field).
*   **Validation**: Double-pass Zod-style schema check on the device before server submission.
*   **Quick Biometric Sign-In**: Securely logs users in using finger-sensor scans (integrated with Android Biometric API) to bypass typing password, storing token hashes in secure key chains.

### B. Device & Session Integrity
1.  API exchanges credentials for `accessToken` and `refreshToken`.
2.  Store tokens locally in **EncryptedSharedPreferences**.
3.  **Multi-Device Handling**: If the partner logs in on another device, the previous token is revoked, prompting a secure logout on the old device.

## 4. Screen-by-Screen Breakdown (Mobile Layout)

### Screen 1: Partner Dashboard
*   **Top Welcome Header**: Show Partner Business Name and active zone (e.g., "Solar Solutions Ltd - North Zone").
*   **Core Performance Stats Grid (Three Cards)**:
    *   `Active Projects`: Count of customer referrals currently in progress.
    *   `Total Earnings`: Total commission value generated by completed installs (e.g., `₹4,50,000`).
    *   `Pending Payout`: Outstanding commissions awaiting admin release or in withdrawal review (e.g., `₹75,000`).
*   **Quick Action Row**:
    *   Button: `+ Refer Client` (launches client intake modal).
    *   Button: `₹ Request Payout` (launches payment withdrawal dialog).
*   **Recent Activity List**: Lists the latest 3 customer referral status changes (e.g., "Rajesh Kumar moved from Site Survey to Installation").

### Screen 2: Refer a Client (Intake Form)
*   A clean, multi-step vertical intake form optimized for quick inputs in the field.
*   **Fields**:
    *   `Customer Full Name` (Input validation: minimum 2 chars).
    *   `Phone Number` (Numeric only validation).
    *   `Email Address` (Optional, generates a placeholder if left blank).
    *   `System Size (kW)` (Dropdown or step-stepper: 1kW, 3kW, 5kW, 10kW etc.).
    *   `City` & `Full Address` (With auto-complete search using Google Places SDK).
*   **Offline Support**: If offline, saves details locally with a "Pending Upload" status, notifying the partner that the lead will sync once they hit network coverage.

### Screen 3: My Referrals & Project List
*   Searchable list of all customers referred by the partner.
*   **Individual Project Row Details**:
    *   Customer name and project address.
    *   System size (e.g., `5.0 kW`) and estimated contract value.
    *   **Visual Stage Tracker Progress Bar**:
        *   `Lead` (25% progress)
        *   `Site Survey` (50% progress)
        *   `Installation` (75% progress)
        *   `Complete` (100% progress)
    *   **Commission Badge**: Est. commission amount (e.g., `₹12,500`) and payment status label (`paid` in green, `pending` in amber). Clicking `View Proof` opens a web viewer to display the bank transfer receipt uploaded by the admin.

### Screen 4: Earnings Ledger & Bank Payouts
*   **Available Balance Panel**: Large digital card showing real-time withdrawable funds. Clicking "Withdraw Funds" triggers a bottom-sheet.
*   **Withdrawal Request Sheets**:
    *   Select Registered Bank Account.
    *   Input Withdrawal Amount.
    *   Click "Request Transfer" (Pushes request to endpoint, notifies admin instantly).
*   **Payout Ledger Table**:
    *   Detailed entries sorting credits (commission earned per project) and debits (bank payouts).
    *   Columns: Date, Description, Reference (Customer Code or Transaction ID), Amount, and Status.
    *   **Receipt Viewer**: If payment status is complete, partners can view the transfer proof image.

### Screen 5: Support Live Chat
*   Direct communication channel with the SWAYOG Support Coordinator/Super Admin.
*   **Features**:
    *   Persistent WebSocket or long-polling HTTP route to transmit messages instantly.
    *   Message bubble interface showing sender status, avatar, message body, and timestamps.
    *   "Typing" and "Message Read" status markers.
    *   Offline cache: Allows viewing messages history when offline and queuing outgoing messages.
    *   Push notifications: Uses Firebase Cloud Messaging (FCM) to trigger high-priority alerts when support replies, even if the app is closed.

## 5. API & Database Sync Mechanism

The partner app operates on an **Offline-First Synchronization** logic, prioritizing local database operations to ensure high performance.

```
       [Partner App UI]
              ^
              | (Reactive Flow / LiveData)
              v
     [Local Database SQLite] <--- (Writes Leads instantly)
              ^
              | (WorkManager Syncs when Online)
              v
      [Retrofit Client]
              ^
              | (API Endpoint Requests)
              v
      [SWAYOG Server API]
```

### A. SQLite/Room Entity Schema
*   **`PartnerProfile`**: Stores user ID, business name, service zone, bank credentials cache.
*   **`ReferredCustomer`**: Stores referrals, system size, installation stage, commission, and local sync state (`isSynced`).
*   **`PayoutLedger`**: Stores credit/debit records, bank receipts, and transaction dates.
*   **`MessageLog`**: Stores chat history.

### B. Synchronization Rules
1.  **Lead Syncing**:
    *   Leads submitted offline are saved with `isSynced = false` in `ReferredCustomer`.
    *   A persistent worker checks for internet. Once online, it sends the data to `/api/v1/partner/requests` (which is mapped to customer registrations) and updates the local ID and `isSynced = true`.
2.  **Ledger and Stats Sync**:
    *   Stats are fetched from `/api/v1/partner/stats`.
    *   Ledger entries are built dynamically by merging the customers list (credits) and partner payouts (debits).
3.  **Chat Log Sync**:
    *   Chat logs are polled or synced via `/api/v1/messages/` endpoints and stored locally.

### C. API Endpoint Mapping Reference

| Action Event | HTTP Method | Endpoint | Request Body / Query Params |
|---|---|---|---|
| User Authentication | `POST` | `/api/v1/auth/login` | `{ "loginId", "password" }` |
| Token Refresh | `POST` | `/api/v1/auth/refresh` | `{ "refreshToken" }` |
| Get Partner Profile | `GET` | `/api/v1/partner/profile` | Header: `Authorization: Bearer <token>` |
| Fetch Partner Stats & Payouts | `GET` | `/api/v1/partner/stats` | Header: `Authorization: Bearer <token>` |
| Register Customer Referral | `POST` | `/api/v1/customers` | `{ "fullName", "phoneNumber", "email", "city", "address", "systemSizeKw", "installationDate", "status": "active", "projectStage": 1 }` |
| Get Referred Customers List | `GET` | `/api/v1/customers` | Header: `Authorization: Bearer <token>` |
| Submit Payout Request | `POST` | `/api/v1/partner/requests` | `{ "serviceType": "Payout Request", "description", "preferredDate" }` |
| Fetch Chat Messages | `GET` | `/api/v1/messages/:adminId` | Header: `Authorization: Bearer <token>` |
| Send Chat Message | `POST` | `/api/v1/messages` | `{ "receiverId", "content" }` |

## 6. Safety, Security & Performance Guidelines

*   **Financial Security**: Any bank account changes or payouts request require biometric validation (fingerprint/PIN verification) to prevent unauthorized transfers if the phone is stolen.
*   **Request Rate Limiting**: The app throttles retry requests on failures (using an exponential back-off algorithm) to avoid overload on the backend PostgreSQL server.
*   **Data Minimization**: The client lists only customers mapped to the specific `partnerId`. The backend enforces complete data isolation via Prisma filters, and the mobile app validates that no unauthorized project data is synced locally.
*   **Asset Management**: Images of payment proofs (receipts) are cached locally in the app's external files directory to avoid fetching raw URLs repeatedly, saving cellular data.
